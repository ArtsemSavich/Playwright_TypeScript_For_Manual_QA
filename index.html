The Terminal

The terminal is a text interface to your computer. Instead of clicking icons, you type commands. You will use it to run TypeScript files, install packages, and execute tests. You do not need to master it — you need to be comfortable with 4 things:

  • Opening the terminal — Terminal.app on Mac, PowerShell or Windows Terminal on Windows
  • Navigating folders: cd folder-name to go into a folder, cd .. to go back, ls (Mac) or dir (Windows) to see what is inside
  • Running a command: type it and press Enter
  • Reading output: most commands print a result or an error — read it before asking for help

Practice now:

Try these 5 commands bash
1. Open your terminal
2. cd Desktop                    # go to Desktop folder
3. mkdir qa-automation           # create a new folder
4. cd qa-automation              # go inside it
5. ls                            # (Mac) or dir (Windows) — folder is empty

If you completed these 5 steps without errors, you are ready.

Create a GitHub Account

GitHub is where your code lives online. You need an account before configuring Git identity — your Git email must match your GitHub email, otherwise your commits will not show up in your GitHub profile.

  • Go to github.com/signup and create a free account (2 minutes)
  • Verify your email — GitHub will send a confirmation link
  • Recommended: use your @solvd.com email — your work on this learning path will be tied to your Solvd identity, which is useful if you want to reference it later (e.g. in a performance review or portfolio). You can always add personal emails to the same GitHub account later via Settings → Emails.
  • Pick a username you are comfortable with professionally — it will show up on your PRs and commits
🔐
Enable 2FA (two-factor authentication) — GitHub requires it. After creating your account, go to Settings → Password and authentication → Two-factor authentication. Options:
  • Authenticator app (recommended) — Google Authenticator, 1Password, Authy, Microsoft Authenticator
  • SMS — works but less secure
Save your backup codes somewhere safe (password manager, printed paper). If you lose your phone and don't have backup codes, recovering the account is painful.
💡
About passwords: GitHub does not accept your password when pushing code from the terminal. You need either a Personal Access Token, an SSH key, or — the easiest option — the GitHub CLI (two sections below). We will use the CLI so you never have to deal with tokens.

Authenticate with GitHub

Run this once. It opens your browser, you log in to GitHub, done.

bash
gh auth login

You will see a series of prompts. Choose these answers:

  • Where do you use GitHub? → GitHub.com
  • Preferred protocol for Git operations? → HTTPS
  • Authenticate Git with your GitHub credentials? → Yes
  • How would you like to authenticate? → Login with a web browser

Copy the one-time code shown in the terminal, press Enter, a browser tab opens — paste the code, approve (2FA prompt may appear). You are done.

Verify:

bash
gh auth status
# ✓ Logged in to github.com as YOUR-USERNAME

Connect git to your GitHub credentials

The previous step logged in the gh CLI only. To make plain git push / git pull work without asking for a password, run one more command:

bash
gh auth setup-git

This configures Git's credential helper to pull your token from gh automatically. No output means success.

✅
What this unlocks: after both commands above (gh auth login + gh auth setup-git), every git push, git pull, and gh repo create will work without passwords or tokens. You will not think about authentication again.

TypeScript Features You Get for Free

VS Code has built-in TypeScript support. These features save you time, especially as a non-developer.

Theory

IntelliSense — autocomplete while you type

Start typing page. in a test file. A dropdown appears with every method Playwright offers: goto, click, fill, getByRole. Pick one with arrow keys + Enter (or Tab).

This means: you do not need to memorize API methods. Let VS Code show them to you.

Theory

Hover for documentation

Hover your mouse over any Playwright method (e.g. getByRole) in your test file. A tooltip appears showing what it does, what parameters it takes, and usually an example. No need to open the docs.

Theory

Go to Definition (F12)

Place your cursor on a class name, e.g. LoginPage, and press F12. VS Code jumps to where LoginPage is defined. Press Ctrl+- (Mac: ⌃+-) to go back.

Use this when you see a method being called and want to see how it works.

Theory

Red squiggles — TypeScript errors

If you see a red wavy underline, TypeScript found a problem. Hover over it to read the message. Common examples:

  • "Cannot find name 'expect'" — you forgot to import it.
  • "Property 'fil' does not exist on type" — typo; you meant fill.
  • "Expected 1 arguments, but got 0" — a method needs a parameter you did not pass.

Fix these before running tests — they are 5-second fixes, and ignoring them means the test will not even start.

TOPICS

Theory

Manual vs Automated testing

  • Why automate? Regression testing is repetitive. Automation runs the same checks every time, faster, without human error.
  • What is Playwright? A tool by Microsoft that controls a real browser through code. You write instructions like "go to this URL", "click this button", "check if this text appears".
  • What is E2E testing? End-to-end tests simulate a real user journey: open the app, log in, perform actions, verify results.
  • When NOT to automate: exploratory testing, one-time checks, tests that change every sprint, highly visual UI checks.
Theory

Understanding playwright.config.ts — the 3 things you care about now

playwright.config.ts (simplified) typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',

  use: {
    // 1. baseURL — the site your tests run against.
    // When set, you can write page.goto('/login') instead of
    // page.goto('https://practicetestautomation.com/login')
    baseURL: 'https://practicetestautomation.com',

    // 2. trace — record execution for debugging
    trace: 'on-first-retry',
  },

  // 3. projects — which browsers to run tests in.
  // By default, EACH test runs in all 3 browsers.
  // That's why running 1 test shows "3 tests passed" in the report.
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit',   use: { ...devices['Desktop Safari'] } },
  ],
});
💡
Important: baseURL is what lets you write page.goto('/login') in tests. Without it, relative URLs fail. You will set this in Week 4 for your login test.

TOPICS

Theory

Variables, types, and strings

typescript
// Variables
const username = "john@test.com";   // const = value will not change
let attempts = 0;                    // let = value can change later
attempts = 3;                        // OK — let allows reassignment

// Types — TypeScript knows what type each value is
const name: string = "Alice";        // text
const age: number = 30;              // number
const isAdmin: boolean = true;       // true or false

// Template strings — inserting variables into text
const greeting = `Hello, ${name}! You are ${age} years old.`;

// Arrays — lists of values
const browsers: string[] = ["chromium", "firefox", "webkit"];

Key takeaway: const = fixed, let = changeable. The : string part tells TypeScript what kind of data the variable holds.

Theory

Functions and async/await

Two common ways to declare a function in TypeScript:

Option 1: function keyword typescript
function greet(name: string): string {
  return `Hello, ${name}`;
}

greet("Alice");   // "Hello, Alice"
Option 2: arrow function (shorter syntax, same result) typescript
const greetShort = (name: string): string => `Hello, ${name}`;

greetShort("Alice");   // "Hello, Alice"

Now the important one — async/await:

typescript
// async/await — "wait for this to finish before continuing"
// Playwright uses this EVERYWHERE because browser actions take time
async function login(page) {
  await page.goto("https://example.com/login");  // wait for page load
  await page.fill("#email", "user@test.com");     // wait for typing
  await page.click("button[type=submit]");        // wait for click
}

// Without "await", the test would try to click
// before the page even finishes loading!

Key takeaway: Every Playwright action starts with await. It means "do not go to the next line until this is done."

Theory

typescript
// Import from an installed package (lives in node_modules/)
import { test, expect } from "@playwright/test";

// Import from your own file (relative path — starts with ./)

Anatomy of a Playwright test file

example.spec.ts typescript
import { test, expect } from "@playwright/test";

test.describe("Login page", () => {

  test.beforeEach(async ({ page }) => {
    await page.goto("https://example.com/login");
  });

  test("should show error for wrong password", async ({ page }) => {
    await page.fill("#email", "user@test.com");
    await page.fill("#password", "wrongpassword");
    await page.click("button[type=submit]");

    await expect(page.locator(".error-message")).toBeVisible();
    await expect(page.locator(".error-message")).toHaveText("Invalid credentials");
  });
});
KeywordWhat it means
import { test, expect }Load tools from the Playwright library
test.describe("name", ...)Group related tests (like a folder)
test.beforeEachRuns before EVERY test — good for setup
test("name", async ({ page }) => ...)One test case. page = your browser tab.
await"Wait for this to finish"

Locators — Finding Elements on the Page

Estimated time: 6-7 hours
0%done

LEARNING OBJECTIVES

  • Find any element on a page using Playwright locators
  • Choose the best locator type for each situation
  • Use Playwright Inspector and codegen to discover locators
  • Chain and filter locators when multiple similar elements exist
⚠️
This is the #1 skill for QA automation. Bad locators = flaky tests that fail randomly. Good locators = stable tests that only fail when there is a real bug. Spend extra time here.

TOPICS

Theory

What is a locator?

A locator tells Playwright "find this element on the page." Think of it like directions: "the big red button at the top of the form."

typescript
// Different ways to find the same "Login" button:

page.getByRole("button", { name: "Login" })     // By role (BEST)
page.getByTestId("login-button")                 // By test ID (GREAT)
page.getByText("Login")                          // By text (GOOD)
page.getByPlaceholder("Enter your email")        // By placeholder (GOOD)
page.getByLabel("Password")                      // By label (GOOD)
page.locator("button.btn-primary")               // By CSS (OK)
page.locator("#login-btn")                       // By ID (OK)
page.locator("//div[@class='form']/button[1]")   // By XPath (AVOID)
Reference

Locator priority: best to worst

PriorityLocatorExampleWhy
1stgetByRolepage.getByRole("button", { name: "Submit" })How a real user thinks — "click the Submit button"
1stgetByTestIdpage.getByTestId("submit-btn")Developers add data-testid specifically for tests
2ndgetByTextpage.getByText("Welcome back")Readable, but breaks if text changes
2ndgetByPlaceholderpage.getByPlaceholder("Email")Great for input fields
2ndgetByLabelpage.getByLabel("Password")Uses form labels — accessible and stable
3rdCSS selectorpage.locator(".btn-primary")CSS classes are meant for styling, not testing
LastXPathpage.locator("//div[2]/button")Breaks if anything in HTML structure changes
Theory

Using Playwright Inspector to find locators

bash
# Open codegen — records your clicks as code
npx playwright codegen https://your-site.com

A browser opens alongside a recording window. Click around your website and Playwright generates test code with locators automatically.

💡

Theory

Chaining and filtering — multiple similar elements

typescript
// Problem: 3 "Delete" buttons on the page
// Solution: narrow down by parent

page.locator(".user-settings").getByRole("button", { name: "Delete" })

page.locator("tr").first().getByRole("button", { name: "Edit" })

page.getByRole("listitem").filter({ hasText: "Product A" })

page.locator("[data-testid='cart-item']")
    .filter({ hasText: "Running Shoes" })
    .getByRole("button", { name: "Remove" })

export default defineConfig({

TOPICS — WRITING TESTS

Reference

Browser actions cheat sheet

typescript
// Navigation
await page.goto("https://example.com");
await page.goto("/login");           // uses baseURL from config
await page.goBack();
await page.reload();

// Clicking
await page.getByRole("button", { name: "Save" }).click();
await page.getByText("Learn more").click();

// Typing
await page.getByPlaceholder("Email").fill("test@example.com");
await page.getByRole("textbox").clear();

// Dropdowns
await page.getByLabel("Country").selectOption("US");

// Checkboxes
await page.getByLabel("I agree to terms").check();

// Keyboard
await page.keyboard.press("Enter");

// Waiting (only when auto-wait is not enough)
await page.waitForURL("**/dashboard");
await page.locator(".spinner").waitFor({ state: "hidden" });
🚫
Never use page.waitForTimeout(3000). It pauses the test for a fixed time regardless of what the page is doing. This is the #1 cause of flaky tests. Playwright's assertions (expect(...).toBeVisible()) already wait automatically — use them instead.
Reference

Assertions cheat sheet

typescript
await expect(page.getByText("Welcome")).toBeVisible();
await expect(page.getByText("Error")).not.toBeVisible();
await expect(page.getByTestId("username")).toHaveText("John Doe");
await expect(page.getByTestId("message")).toContainText("success");
await expect(page).toHaveURL(/.*dashboard/);
await expect(page).toHaveTitle("Dashboard - MyApp");
await expect(page.getByLabel("Email")).toHaveValue("test@example.com");
await expect(page.getByRole("listitem")).toHaveCount(5);
await expect(page.getByRole("button", { name: "Submit" })).toBeEnabled();
await expect(page.getByRole("button", { name: "Submit" })).toBeDisabled();

Tip: Add a failure message to assertions: expect(locator, 'Error should be visible after wrong password').toBeVisible()

Debugging process — use this every time

  1. Read the error message fully. What did it expect? What did it get?
  2. Open the HTML report. Look at the screenshot of the moment it failed.
  3. Open the trace. Step backwards from the failure.
  4. If still unsure: run with --headed or --ui and watch the browser.
  5. Add page.pause() just before the failure to inspect the page live.

NEW TOPIC — PAGE OBJECT MODEL

Theory

Page Object Model (POM) — define locators once, reuse everywhere

If a button locator appears in 5 tests and the button changes, you update 5 files. With a Page Object you define each locator once in a separate file.

pages/LoginPage.ts typescript
import { Page, Locator } from "@playwright/test";

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByPlaceholder("Email");
    this.passwordInput = page.getByPlaceholder("Password");
    this.loginButton = page.getByRole("button", { name: "Login" });
    this.errorMessage = page.locator(".error-message");
  }

  async goto() {
    await this.page.goto("/login");
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }
}

Where to save: Create a pages/ folder next to tests/. One file per page of your app.

Hands-on

Using a Page Object in a test

tests/login.spec.ts typescript
import { test, expect } from "@playwright/test";
import { LoginPage } from "../pages/LoginPage";

test.describe("Login", () => {
  let loginPage: LoginPage;

  test.beforeEach(async ({ page }) => {
    loginPage = new LoginPage(page);
    await loginPage.goto();
  });

  test("successful login", async ({ page }) => {
    await loginPage.login("user@company.com", "password123");
    await expect(page).toHaveURL(/.*dashboard/);
  });

  test("invalid password shows error", async () => {
    await loginPage.login("user@company.com", "wrong");
    await expect(loginPage.errorMessage).toBeVisible();
  });
});
💡
Notice how clean the tests are now. If the login button changes, you update ONE file (LoginPage.ts) instead of every test.

Common Pitfalls

These are the most frequent mistakes when learning automation for the first time. Knowing them saves time.

PitfallWhy It HappensFix
Skipping the TypeScript weekPlaywright looks like the real workWeek 2 is not optional. Weak TypeScript causes 80% of test bugs.
Not using awaitEasy to forget — the code still runs, just wronglyRule: every Playwright method call needs await. No exceptions.
Using CSS selectors instead of semantic locatorsThey look familiar from DevToolsTry getByRole and getByLabel first. They break less often when the UI changes.
Using page.waitForTimeout()Feels like an easy fix for timing issuesNEVER use it. Use auto-waiting assertions like expect(...).toBeVisible().
Guessing instead of reading the errorErrors feel intimidatingRead the full error message. It usually tells you exactly what is wrong.
Tests that only work sometimesRace condition or hard waitsReplace waitForTimeout with expect(...).toBeVisible().
Writing all tests in one fileFeels simpler at firstOne spec file per feature or flow. Easier to navigate and debug.
Forgetting test.only in a PRUsed for debugging, forgot to removeCheck your diff before committing. Your CI will run only one test otherwise.
