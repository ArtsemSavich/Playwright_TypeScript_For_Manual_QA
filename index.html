The Terminal

The terminal is a text interface to your computer. Instead of clicking icons, you type commands. You will use it to run TypeScript files, install packages, and execute tests. You do not need to master it — you need to be comfortable with 4 things:

  • Opening the terminal — Terminal.app on Mac, PowerShell or Windows Terminal on Windows
  • Navigating folders: cd folder-name to go into a folder, cd .. to go back, ls (Mac) or dir (Windows) to see what is inside
  • Running a command: type it and press Enter
  • Reading output: most commands print a result or an error — read it before asking for help

Practice now:

Try these 5 commands bash
1. Open your terminal
2. cd Desktop                    # go to Desktop folder
3. mkdir qa-automation           # create a new folder
4. cd qa-automation              # go inside it
5. ls                            # (Mac) or dir (Windows) — folder is empty

If you completed these 5 steps without errors, you are ready.

The previous step logged in the gh CLI only. To make plain git push / git pull work without asking for a password, run one more command:

This configures Git's credential helper to pull your token from gh automatically. No output means success.

TypeScript Features You Get for Free

VS Code has built-in TypeScript support. These features save you time, especially as a non-developer.

Theory

IntelliSense — autocomplete while you type

Start typing page. in a test file (use the .spec.ts extension, e.g. example.spec.ts). A dropdown appears with every method Playwright offers: goto, click, fill, getByRole. Pick one with arrow keys + Enter (or Tab).

This means: you do not need to memorize API methods. Let VS Code show them to you.

Theory

Hover for documentation

Hover your mouse over any Playwright method (e.g. getByRole) in your test file. A tooltip appears showing what it does, what parameters it takes, and usually an example. No need to open the docs.

Theory

Go to Definition (F12)

Place your cursor on a class name, e.g. LoginPage, and press F12. VS Code jumps to where LoginPage is defined. Press Ctrl+- (Mac: ⌃+-) to go back.

Use this when you see a method being called and want to see how it works.

Theory

Red squiggles — TypeScript errors

If you see a red wavy underline, TypeScript found a problem. Hover over it to read the message. Common examples:

  • "Cannot find name 'expect'" — you forgot to import it.
  • "Property 'fil' does not exist on type" — typo; you meant fill.
  • "Expected 1 arguments, but got 0" — a method needs a parameter you did not pass.

Fix these before running tests — they are 5-second fixes, and ignoring them means the test will not even start.

TOPICS

Theory

Manual vs Automated testing

  • Why automate? Regression testing is repetitive. Automation runs the same checks every time, faster, without human error.
  • What is Playwright? A tool by Microsoft that controls a real browser through code. You write instructions like "go to this URL", "click this button", "check if this text appears".
  • What is E2E testing? End-to-end tests simulate a real user journey: open the app, log in, perform actions, verify results.
  • When NOT to automate: exploratory testing, one-time checks, tests that change every sprint, highly visual UI checks.

Theory

What's inside tests/example.spec.ts

When you ran npm init playwright@latest, Playwright created an example test file for you. This is the test you'll be running in the exercises. Here's exactly what it contains:

tests/example.spec.ts typescript
import { test, expect } from '@playwright/test';

test('has title', async ({ page }) => {
  await page.goto('https://playwright.dev/');

  // Expect a title "to contain" a substring.
  await expect(page).toHaveTitle(/Playwright/);
});

test('get started link', async ({ page }) => {
  await page.goto('https://playwright.dev/');

  // Click the get started link.
  await page.getByRole('link', { name: 'Get started' }).click();

  // Expects page to have a heading with the name of Installation.
  await expect(page.getByRole('heading', { name: 'Installation' })).toBeVisible();
});

Two tests in one file. In plain English:

  • Test 1 — "has title": opens playwright.dev, checks that the page title contains the word "Playwright".

PART 4 — DESTRUCTURING

Theory

Pulling values out of objects

You see this pattern in every Playwright test:

typescript
// This is destructuring — pulling "page" out of the object Playwright gives you
test("example", async ({ page }) => {
  await page.goto("https://example.com");
});

// Same idea with your own data:
const user = { email: "john@test.com", password: "Secret123" };

// Instead of writing user.email and user.password everywhere:
const { email, password } = user;
// Now: email = "john@test.com", password = "Secret123"

// Use directly
await page.getByPlaceholder("Email").fill(email);
await page.getByPlaceholder("Password").fill(password);

Key takeaway: { page } in test functions is not special syntax — it is standard destructuring. The curly braces mean "take the page property out of the object."

PART 6 — ASYNC/AWAIT IN PLAYWRIGHT

Theory

Why browser actions need await

Browser actions take time: loading a page, clicking a button, typing text. await means "do not go to the next line until this action is done."

typescript
import { Page } from "@playwright/test";

async function login(page: Page): Promise<void> {
  await page.goto("https://example.com/login");     // wait for page to load
  await page.getByPlaceholder("Email").fill("user@test.com");  // wait for typing
  await page.getByRole("button", { name: "Login" }).click();   // wait for click
}

// Without "await", the test would try to click
// before the page even finishes loading!
PartWhat it means
asyncMarks the function as asynchronous — it can use await inside
await"Wait for this to finish before continuing"
Promise<void>The function returns a Promise and produces no value. async functions always return a Promise; await waits for it.
PageThe Playwright type for a browser tab — imported from @playwright/test
⚠️
The #1 beginner mistake: forgetting await. The code will still run without errors — but it will do things in the wrong order and fail randomly. Rule: use await with Playwright actions (.click(), .fill(), .goto()) and assertions (expect). Locators like page.getByRole(...) do not need await by themselves — only the action on them does.

PART 7 — COMMON TYPESCRIPT ERRORS AND HOW TO FIX THEM

Theory

Error #1: string | undefined

This is the most common TypeScript error you will see. It means: "this value might be a string, or it might not exist at all."

The error text
Argument of type 'string | undefined' is not assignable
to parameter of type 'string'.
Fix: check before using typescript
// Problem: promoCode might be undefined
function applyPromoCode(code: string | undefined) {
  // console.log(code.toUpperCase());  // ERROR — code might be undefined!

  // Fix: check first, then use
  if (code) {
    console.log(code.toUpperCase());  // OK — TypeScript knows code is string here
  }
}

What happens: string | undefined means the value can be a string OR it can be missing. Before you use it as a string, you need to check. After if (code), TypeScript "narrows" the type and knows it is definitely a string.

Theory

Error #2: "Property does not exist"

The error text
Property 'fullName' does not exist on type '{ email: string; password: string; }'.
Why it happens typescript
const user = { email: "john@test.com", password: "Secret123" };

console.log(user.fullName);  // ERROR — there is no "fullName" property

// Fix: use only properties that exist, or add the missing one
console.log(user.email);  // OK

This is TypeScript protecting you from typos and missing data. Read the error carefully — it tells you exactly which property is missing and on which object.

Theory

Error #3: "Argument is not assignable"

The error text
Argument of type 'number' is not assignable to parameter of type 'string'.
Why it happens typescript
function fillField(value: string) {
  console.log(value);
}

fillField(42);              // ERROR — number is not a string
fillField("42");            // OK — "42" is a string
fillField(String(42));      // OK — converted to string

TypeScript checks that you pass the right type. This catches real bugs: you would not want to type a number into a text field without converting it first.

Reference

Optional chaining ?. — safe access to nested data

typescript
type ApiResponse = {
  error?: {
    message?: string;
  };
};

const response: ApiResponse = {};

// Without ?. — crashes if response.error is undefined
// const message = response.error.message;  // ERROR!

// With ?. — returns undefined instead of crashing
const message = response.error?.message;   // undefined, no crash

Rule: if the thing on the left might be null or undefined, use ?. to avoid a crash. You will see this in API responses, parsed JSON, and debug helpers.

Theory

1. Modern UI is dynamic

UI is not static HTML — it changes constantly:

  • Elements appear after API calls complete
  • Buttons change state (enabled → disabled → enabled)
  • Content updates after user actions (e.g., cart badge counter)
⚠️
Your test must wait for the right state, not just execute steps. A test that clicks a button before the page has loaded will fail — not because the feature is broken, but because the test is wrong.
Theory

2. Element states (critical concept)

Before interacting with an element, it must be in the correct state:

StateMeaningExample
Visible / HiddenThe element is rendered and not coveredA modal overlay hides the form behind it
Enabled / DisabledThe element accepts user interactionSubmit button disabled until fields are filled
Attached / DetachedThe element exists in the DOMA spinner removed from DOM after data loads
⚠️
Most test failures happen because you interact with an element too early — before it is visible, enabled, or stable. Playwright’s auto-waiting helps, but you must understand what it waits for.
Theory

3. Playwright auto-waiting

When you call .click(), .fill(), or any action, Playwright automatically waits until the element is:

  • Attached to the DOM
  • Visible
  • Stable (not animating)
  • Able to receive events (not blocked by another element)
  • Enabled (for buttons and inputs)
⚠️
Auto-wait does NOT cover everything. Playwright does not wait for:
  • Data to finish loading from an API
  • Application logic to complete (e.g., cart recalculation)
  • URL to change after a click
For these, use explicit assertions: await expect(page).toHaveURL(/inventory/) or await expect(locator).toBeVisible().

Theory

6. How to validate a locator like a QA

A) Key principle

Pick Locator is a starting point, not a final solution. The VS Code extension generates locators quickly, but you must always validate them before using in production tests.

B) Why this matters

Bad locator = flaky test. If your locator breaks because the app changed, your test breaks. The app might work fine, but the test fails. This is noise.

C) Step-by-step validation process
  1. Generate locator (Pick Locator) — Use VS Code extension to create a first draft
  2. Open DevTools — F12 (Windows/Linux) or ⌘+⌥+I / fn+F12 (Mac) → go to the Elements tab
  3. Inspect element — Find your element in the HTML and check its attributes, role, text
  4. Look for stable attributes:
    • data-testid — added by developers for testing
    • aria-label — accessible label for screen readers
    • role + name — semantic HTML (e.g., button with text)
    • id — usually stable, but check if it's unique
  5. Avoid unstable selectors:
    • Dynamic CSS classes (change on page updates)
    • Deep nesting (break when layout changes)
    • Indexes (break when list order changes)
  6. Check uniqueness — In DevTools Elements tab, press ⌘+F (Mac) or Ctrl+F (Windows) to open the search bar. It supports CSS selectors and XPath:
    css
    # CSS selector example (type directly in Elements search bar):
.inventory_item

# Result: "3 of 6" — means 6 matches found (not unique!)
# Result: "1 of 1" — unique, good locator
    xpath
    # XPath example (prefix with //  in Elements search bar):
//button[@data-testid='login-button']

# XPath by text:
//button[contains(text(), 'Login')]

# Result shown as: "1 of 1" (unique) or "1 of 3" (not unique)

    Read the result (shown in top-right of search bar):

    • 1 of 1 = good (unique locator)
    • 1 of N (N > 1) = bad (too many matches, not specific enough)
    • No results = wrong (locator does not match anything)
  7. Simplify locator — Use the most readable version:
    typescript
    // ❌ Bad — fragile and hard to read
page.locator("div > div > form > button.btn-primary:nth-child(3)")

// ✓ Better — semantic and stable
page.getByRole("button", { name: "Login" })
D) Single vs Multiple elements

Single element: Use when you expect exactly one match.

typescript
// Expect one button on page
await expect(page.getByRole("button", { name: "Login" })).toBeVisible();

Multiple elements (list): Use when selecting from a group.

typescript
// Select all products in list (returns locator for the group)
const products = page.locator(".inventory_item");

// Get count
const count = await products.count();

// Click first product
await products.first().click();

// Click the 3rd product
await products.nth(2).click(); // nth is 0-indexed

// Click by text
await products.filter({ hasText: "Sauce Labs Backpack" }).click();

Rule: Use index (`.nth()`) only if the order is stable and important. Better: filter by text or data-testid.

E) Working with developers

If there is no stable locator — ask developers to add one. Do not try to make a flaky test work. Make the element testable instead.

F) XPath: when to use it, when to avoid it

Real talk: XPath is used heavily in real projects. It's a powerful tool when used correctly, but fragile when used badly. Here's an honest breakdown.

Why XPath is popular:

  • Universal — works on any website, any app, any structure
  • Flexible — can select by position, attribute, text, parent, sibling, complex logic
  • No test attributes required — works even if developers did not add data-testid
  • Complex queries — can find elements others cannot (by parent, ancestor, following sibling)

Problems with XPath:

  • Fragile when used poorly — bad XPath breaks on any HTML change
  • Hard to read — future QA will struggle to understand it
  • Slow performance — complex XPath queries are slower than CSS selectors
  • Not Playwright's recommendation — but only when you have better options

Bad XPath examples (fragile):

xpath
// ❌ Depends on position — breaks if order changes
//div[3]/button

// ❌ Deep nesting — breaks on layout changes
//div/div/div/div/form/div/button

// ❌ Text matching is exact — breaks on copy changes
//button[text()='Click here']

// ❌ Dynamic index — unpredictable
//*[@class='active'][1]

Good XPath examples (stable):

xpath
// ✓ By stable attribute
//button[@data-testid='login-button']
//input[@name='username']
//a[@id='home-link']

// ✓ By partial text match (flexible)
//button[contains(text(), 'Login')]
//span[contains(@class, 'error-message')]

// ✓ By role + accessible name (semantic)
//button[contains(@aria-label, 'Close')]

// ✓ Smart parent traversal (when needed)
//input[@name='password']/parent::form//button[@type='submit']

// ✓ By attribute pattern
//div[starts-with(@id, 'product-')]
//input[contains(@placeholder, 'Email')]

When to use XPath (OK cases):

  • Element has stable data-testid or id — use XPath as fallback
  • Need to find by parent/ancestor relationship — XPath is better than CSS
  • Text matching with partial content — contains() is useful
  • Legacy app with no semantic attributes — sometimes XPath is the only option

When NOT to use XPath (better alternatives exist):

  • Semantic element with role — use getByRole() instead
  • By text content — use getByText() or getByLabel()
  • By visible label — use getByPlaceholder() or getByAltText()
  • By position in simple list — use .nth() if order is stable

The rule of thumb:

PriorityUse ThisWhy
1stgetByRole()Most accessible, stable by definition
2ndgetByTestId()Stable, intentional for testing
3rdgetByText()Good for buttons, links with visible text
4thCSS selectorWhen Playwright helpers don't work
5thXPathWhen CSS can't do it + stable attributes

In real projects: Most test suites use a mix. You'll have 80% semantic selectors and 20% XPath for complex edge cases. The key is: know why you are using XPath, not just "this selector works".

Page Object Model (POM) — define locators once, reuse everywhere

If a button locator appears in 5 tests and the button changes, you update 5 files. With a Page Object you define each locator once in a separate file.

pages/LoginPage.ts typescript
import { Page, Locator } from "@playwright/test";

export class LoginPage {
  readonly page: Page;
  readonly emailInput: Locator;
  readonly passwordInput: Locator;
  readonly loginButton: Locator;
  readonly errorMessage: Locator;

  constructor(page: Page) {
    this.page = page;
    this.emailInput = page.getByPlaceholder("Email");
    this.passwordInput = page.getByPlaceholder("Password");
    this.loginButton = page.getByRole("button", { name: "Login" });
    this.errorMessage = page.locator(".error-message");
  }

  async goto() {
    await this.page.goto("/login");
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.loginButton.click();
  }
}

Where to save: Create a pages/ folder next to tests/. One file per page of your app.

📖 Official docs: playwright.dev/docs/pom — full Page Object Models guide including how to use them with fixtures.
